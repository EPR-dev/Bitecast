A. How the dashboard is built

Bitecast is three layers stacked on top of MapLibre GL: a data layer of curated place knowledge, an engine layer that turns conditions into a verdict, and a UX layer built around field readability and decision support. Nothing on the server — every byte ships to the browser and runs there.

Place knowledge Data
  • ~8 GeoJSON files — shore, park, access, hazards, POIs, eelgrass, zones, bathy grid.
  • Bathymetric grid — ~55 m cells baked with inverse-distance weighting; each cell carries depth_ft, dist_shore_m, dist_eelgrass_m, bottom_class.
  • Live data — NOAA station 9410170 for tides, Open-Meteo for marine weather, 7-day horizon. Both gracefully degrade to a last-good cache.
  • Personal data — IndexedDB for journal entries and photos. Never leaves the device.
Scoring & decisions Engine
  • Species habitat profile — each of 12 species has a depth gaussian plus shore / eelgrass / channel preferences.
  • Per-cell score — species fit × tide-stage match × time-of-day × season weight. Drives the heat map and the "best spot" pin.
  • Verdict — Go / OK / Skip with plain-language reasons. Decision support, not data dump.
  • Slam logic — themed multi-species challenges (Inshore, Summer Bite, Shore-to-Channel, Night Bite) computed from the journal.
Field-first UI UX
  • Scrubbable timeline — 12-h SVG tide curve with sun bands. Every panel re-scores as you drag.
  • 7-day picker — verdict, heat map, and species suggestions follow the chosen day.
  • Field mode — high-contrast, bigger touch targets for sunlight on a boat deck.
  • Offline-first — Service Worker pre-caches shell, GeoJSON, and tiles. Works without signal.
Single thesis

Anyone can render a tide chart. The value compounds in the place knowledge — hand-curated bathymetry, species habitat profiles, and per-place rules that turn raw data into a verdict a 14-year-old can act on.

B. Scaling to an OnWater-class app

Three discrete fundable phases get you from today's demo to a multi-region marine app. Each phase is additive — nothing built in one phase has to be thrown away in the next.

The three-tier roadmap

Stage Scope Key changes Unlocks
Demo (today) Fiesta Island, single user Static GeoJSON, no auth, IndexedDB for personal data Compelling pitch, free hosting, runs on a USB stick
Single-region prod Mission Bay → all of San Diego Auth + backend, PostGIS tile server, real community, native PWA shell, payments First paying users, real social, push notifications, app-store distribution
OnWater-class Any waterbody, NA → global Multi-region ingestion pipeline, per-region rules engine, marine forecast aggregation, regulation database, multi-mode (fishing + paddle + sail) Network effect, content licensing, B2B (charters, guides), defensible moat

Keep / replace / add

Keep as-is Replace under the hood Add new
Per-species habitat model Static JSON → API + tile server Auth + multi-tenant data model
Per-cell scoring function IndexedDB-only → cloud sync Habitat ingestion pipeline
Verdict engine Hardcoded place → region config Per-region regulation database
Slam & achievement logic Service Worker → native offline Native shell (RN / Capacitor)
Field-mode UX Open-Meteo only → forecast aggregator Subscription plumbing
Anonymous community shape Seeded data → moderated submissions Privacy controls (hex-binning, opt-in)
Highest-leverage next move

Build the habitat-data ingestion pipeline before the backend. A weekend per new region beats a quarter per new region. Once bathymetry + eelgrass + OSM amenities + tide-station lookup are scripted, every new waterbody becomes "config + curate" instead of "rebuild." That pipeline is the actual moat.
